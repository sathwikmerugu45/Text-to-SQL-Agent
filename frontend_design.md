# Frontend Design: Self-Healing Text-to-SQL Agent

This document outlines the UI design, layout, and styling approach used for the frontend of the Text-to-SQL Agent application.

## 1. Technology Stack
- **Framework**: Streamlit (`streamlit==1.32.2`)
- **Language**: Python 3.10+
- **Data Handling**: Pandas (for result tables)
- **HTTP Client**: Requests (for communicating with the FastAPI backend)

## 2. Core Layout & Structure

The frontend is designed to be highly functional yet readable, splitting the interface into two main areas: a **Configuration Sidebar** and a **Main Content Area**.

### 2.1 Configuration Sidebar
The sidebar acts as the control panel for the application.
- **API Settings**: Allows the user to configure the backend API URL (defaults to `http://localhost:8000`).
- **Health Check**: A button to ping the backend (`/health`) to verify database connectivity and the number of indexed ChromaDB tables.
- **Table Explorer**: A button to fetch and display (`/tables`) the currently indexed schema tables.
- **Example Questions**: A list of clickable buttons containing pre-written natural language queries (e.g., "Which genre has the most tracks?"). Clicking these auto-fills the main query input box.

### 2.2 Main Content Area
The central workspace where the user interacts with the agent.
- **Header**: Application title and a brief caption explaining the agent's self-healing capabilities.
- **Input Area**: A large text area for entering natural language queries, featuring the pre-filled text from the sidebar if selected.
- **Controls**: A primary "Run Agent" button and a toggle switch to show/hide the detailed "Reasoning Trace".

## 3. Execution & Result Display

When a query is submitted, the frontend displays a loading spinner ("Agent is thinking...") and makes a POST request to the backend `/query` endpoint. The resulting data is displayed sequentially:

### 3.1 Status Badges
Immediate visual feedback on the outcome:
- **✅ SUCCESS**: Green badge indicating the query succeeded.
- **❌ FAILED AFTER X RETRIES**: Red badge indicating the agent exhausted its retry limit.
- **🔄 Self-healed**: Orange badge shown if the agent encountered an error but successfully fixed its own SQL on a subsequent retry.

### 3.2 Answer & SQL
- **💬 Answer**: The natural language summary generated by the agent's Response Formatter node, displayed in a highlighted info box.
- **🗄️ Generated SQL**: The exact SQL query executed, displayed in a syntax-highlighted code block.

### 3.3 Results Table
If the SQL query returned rows:
- **📊 Results Dataframe**: A Pandas DataFrame rendering the raw rows returned from the SQLite database.
- **⬇ Download CSV**: A button allowing the user to export the query results as a CSV file.

## 4. Reasoning Trace (ReAct Loop Visualization)

If the "Show reasoning trace" toggle is active, the UI renders the agent's internal thought process. This is the core educational feature of the app, visualizing LangGraph's node execution.

Each step in the ReAct (Reasoning and Acting) loop is rendered as a distinct card containing:
1. **Node Name**: The component that executed (e.g., `schema_retriever`, `sql_generator`, `sql_executor`).
2. **💭 Thought**: The agent's internal reasoning for *why* it is taking the action.
3. **⚡ Action**: What the agent actually did (e.g., "Executed: SELECT * FROM...").
4. **👁️ Observation**: The result of the action (e.g., the rows returned, or the specific database error encountered).

## 5. Custom CSS & Theming

The application uses custom HTML/CSS injected via `st.markdown(..., unsafe_allow_html=True)` to override default Streamlit styling and create a cohesive, dark-themed dashboard look (heavily inspired by the Catppuccin color palette).

**Key CSS Classes:**
- `.step-card`: Dark background (`#1e1e2e`), rounded corners, and subtle borders for the reasoning trace cards.
- `.step-node`: Bold, uppercase, purple (`#cba6f7`) text for the node titles.
- `.step-label`: Subdued grey (`#6c7086`) uppercase labels for Thought/Action/Observation headers.
- `.step-text`: Light text (`#cdd6f4`) for the actual trace content.
- `.success-badge` / `.error-badge` / `.retry-badge`: Color-coded pills (Green `#a6e3a1`, Red `#f38ba8`, Orange `#fab387`) with dark text for high contrast status indicators.

## 6. Error Handling
- Network errors (e.g., backend offline) are caught and displayed as clean Streamlit error messages rather than stack traces.
- HTTP errors returned by the FastAPI backend are parsed and displayed directly to the user.
